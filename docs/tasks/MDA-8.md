# MDA-8

Introduce database schema versioning with Alembic and implement the initial
persistent Lead model.

This task establishes a controlled migration workflow and creates the first
production-oriented database schema for the backend system.

The project transitions from:

```text
application can connect to PostgreSQL
```

to:

```text
database structure is managed as versioned code
```

## Scope

- Initialize Alembic migration infrastructure.
- Integrate Alembic with SQLAlchemy metadata.
- Introduce initial Lead ORM model.
- Create first migration for leads table.
- Add migration execution workflow.
- Document migration setup and usage.

## Functional Requirements

### Alembic Initialization

Initialize migration infrastructure:

```bash
alembic init alembic
```

The project must include:

- `alembic.ini`
- migration environment configuration
- migration versions directory

### SQLAlchemy Integration

Alembic must use application metadata for migration autogeneration.

Example:

```python
target_metadata = Base.metadata
```

### Lead Schema

Introduce initial Lead persistence model.

Minimum fields:

| Field | Type |
| --- | --- |
| `id` | UUID / string |
| `name` | string |
| `phone` | string |
| `preferred_contact_channel` | string |
| `status` | string |
| `created_at` | timestamp |

### Migration Generation

The project must support autogenerated migrations:

```bash
alembic revision --autogenerate -m "create lead table"
```

### Migration Execution

Database schema must be applied through:

```bash
alembic upgrade head
```

## Acceptance Criteria

- Alembic is integrated into the project.
- SQLAlchemy metadata is connected to Alembic.
- `leads` table migration is created.
- Migrations apply successfully via:

  ```bash
  alembic upgrade head
  ```

- Migration workflow is reproducible on a clean database.
- Database schema changes are version-controlled in git.
- Documentation explains migration workflow.
