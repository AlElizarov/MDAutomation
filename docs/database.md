# Database

This document describes the current database setup, schema management workflow,
and local recovery commands.

## Runtime

The local Docker environment runs PostgreSQL through Docker Compose.

Service:

```text
db
```

The backend receives its database connection string through `DATABASE_URL`.
In Docker Compose, this value is built from the PostgreSQL settings:

```text
POSTGRES_DB=mda
POSTGRES_USER=mda_user
POSTGRES_PASSWORD=<local-password>
```

Docker Compose connection string:

```text
postgresql://mda_user:<local-password>@db:5432/mda
```

For local host access, use `localhost` instead of `db`:

```powershell
$env:DATABASE_URL = "postgresql://mda_user:<local-password>@localhost:5432/mda"
```

## Persistent Storage

PostgreSQL stores data in the named Docker volume `postgres_data`.

The volume is mounted inside the PostgreSQL container at:

```text
/var/lib/postgresql/data
```

Data survives:

- `docker compose restart`
- `docker compose down` followed by `docker compose up`
- backend container recreation

Stop services without deleting database data:

```powershell
docker compose down --remove-orphans
```

Reset the local database completely:

```powershell
docker compose down --volumes --remove-orphans
```

Only use `--volumes` when you intentionally want to delete local PostgreSQL
data.

## SQLAlchemy Layout

Database code lives under `src/app/db`.

```text
src/app/db/
|-- base.py
|-- session.py
`-- models/
    |-- __init__.py
    `-- lead.py
```

Responsibilities:

| File | Responsibility |
| --- | --- |
| `base.py` | Defines the shared SQLAlchemy declarative `Base` and imports models for metadata registration. |
| `session.py` | Creates the SQLAlchemy engine, session factory, FastAPI DB dependency, and readiness check. |
| `models/lead.py` | Defines the persistent `Lead` ORM model. |

Alembic uses `Base.metadata` to compare application models with the actual
database schema during autogeneration.

## Current Schema

### `leads`

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | string | yes | Primary key. Defaults to a generated UUID string in the ORM model. |
| `name` | string | yes | Lead name. |
| `phone` | string | yes | Lead phone number. |
| `preferred_contact_channel` | string | yes | Requested communication channel. |
| `status` | string | yes | Lead lifecycle status. Defaults to `new` in the ORM model. |
| `created_at` | timestamp | yes | Database-generated creation timestamp. |

## Migrations

Database schema changes are managed with Alembic.

Migration files live in:

```text
alembic/versions/
```

The Alembic runtime configuration lives in:

```text
alembic/env.py
```

The first migration creates the `leads` table:

```text
alembic/versions/20260525_0001_create_leads_table.py
```

Apply migrations locally:

```powershell
$env:DATABASE_URL = "postgresql://mda_user:<local-password>@localhost:5432/mda"
.\scripts\db\migrate.ps1
```

Apply migrations inside Docker Compose:

```powershell
docker compose up -d
docker compose exec backend python -m alembic upgrade head
```

Create an autogenerated migration after changing SQLAlchemy models:

```powershell
.\.venv\Scripts\python.exe -m alembic revision --autogenerate -m "describe change"
```

Check whether models and migrations are in sync:

```powershell
.\.venv\Scripts\python.exe -m alembic check
```

Rollback one migration:

```powershell
.\.venv\Scripts\python.exe -m alembic downgrade -1
```

Rollback all migrations:

```powershell
.\.venv\Scripts\python.exe -m alembic downgrade base
```

Rollback commands can delete tables and data. Run them only against databases
where data loss is acceptable.

## Verification

Run unit tests:

```powershell
.\scripts\dev\test.ps1
```

Run the documentation build:

```powershell
.\scripts\docs\build_docs.ps1
```

Run the Docker smoke test:

```powershell
.\scripts\docker\smoke-docker.ps1
```

The Docker smoke test:

- builds the backend image;
- starts PostgreSQL and the backend;
- waits for `/health`;
- applies `alembic upgrade head` inside the backend container;
- verifies that the `leads` table exists;
- creates a marker record in the `leads` table;
- verifies data survives PostgreSQL restart;
- verifies data survives Docker Compose recreation;
- verifies data survives backend container recreation.
